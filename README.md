# CS 255 DriverPass Portfolio

## DriverPass Project Summary

DriverPass was a project for a client who wanted a better way to help people prepare for driving tests. They needed a system that could handle customer accounts, training packages, online classes, practice tests, driving lessons, reports, and DMV updates. The system also needed to help staff schedule lessons and manage the business without using a bunch of separate manual steps.

## What I Did Well

I think I did a good job sorting out what the system needed to do and connecting those needs to the design. I paid attention to the different users, like customers, secretaries, drivers, the owner, the IT officer, and the DMV system. Each one needed different access and different tools.

I also think the diagrams helped show the system in a clear way. The use case diagram showed who uses the system. The activity and sequence diagrams showed how scheduling works. The class diagram helped organize the main parts of the system.

## What I Would Revise

If I could revise one part, I would go back to the business requirements document and make some of the requirements more specific earlier. I would add clearer details about browser support, database needs, security, login limits, and how the system should handle suspicious activity.

That would make the document stronger. It would also make the design easier to follow later because there would be fewer gaps to fix.

## Interpreting User Needs

I interpreted the user’s needs by looking at what DriverPass wanted the system to do in real use. Customers needed to create accounts, choose packages, take practice tests, and schedule lessons. Staff needed to manage appointments, reports, accounts, and DMV updates. I used those needs to build the diagrams and technical requirements.

User needs matter because the system has to work for the people using it. A design can look fine on paper and still fail if it doesn’t match what users actually need to do. Good software should make the work clearer, easier, and less frustrating.

## My Approach to Software Design

My approach to software design is to slow down first and understand the problem. I start with the client’s goals, then I look at the users, the actions they need to take, and the information the system needs to store. After that, I use diagrams to organize the design before getting into technical details.

In the future, I would keep using interviews, requirements lists, use case diagrams, activity diagrams, sequence diagrams, and class diagrams. I would also check the requirements more closely at the start. Fixing problems early is a lot easier than trying to clean them up after the whole design is already built.
